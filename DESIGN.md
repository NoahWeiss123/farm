## Problem Statement

Robotics foundation models (VLAs) are proliferating — π0, π0.5, Gemini Robotics, GR00T N1, OpenVLA — but each one ships its own SDK, runtime, embodiment assumptions, and operational quirks. A developer who wants to put one of these models on their robot has to learn a new stack per model, host the inference themselves, and pick blind between them.

FARM builds the missing layer: a hosted **agent harness** that sits between "user describes a task" and "robot executes it." The harness exposes a uniform API across multiple VLA backends, hosts inference on Cloudflare's GPU infrastructure, and includes an **auto mode** where an LLM router decomposes the task and dispatches subtasks to whichever model is best suited per subtask. The name is literal: a *farm* of action-reasoning models, with infrastructure to grow, evaluate, and route between them.

The demonstration target is a UFactory 850 6-DOF arm, with a fine-tuned π0.5 as the primary VLA backend and Gemini Robotics proxied as a secondary backend.

## Users

The harness targets four user types, in priority order:

1. **University research labs (primary)** — graduate students and PIs running manipulation experiments. They have mid-range arms (UFactory, Franka, UR5e), no in-house ML platform team, and need to compare VLA baselines without each lab member rebuilding the inference stack. Pain point: every paper reimplements the same harness. The product is best at *making cross-model comparisons reproducible*.
2. **Industrial / manufacturing teams** — small and mid-size manufacturers exploring AI-driven manipulation for tasks where fixed automation is too rigid (kitting, low-volume assembly, defect handoff). They have industrial arms (UFactory 850, UR5/10, KUKA KR-class) and care about reliability, safety envelopes, and auditability above all. The product is best at *deterministic fallback paths and per-action auditability*. This persona is the credibility anchor: if it works for them, it works.
3. **Hobbyists / educators** — people with low-cost arms (SO-ARM100/101, LeRobot rigs) who want to play with frontier VLAs without owning GPUs. They drive the ecosystem flywheel: open-source contributions, dataset sharing, viral demos. The product is best at *zero-setup access to hosted models*.
4. **Robotics startups** — teams building vertical applications who want to defer the "which VLA?" question and focus on their domain. Highest willingness to pay, smallest count. The product is best at *swap-out-the-backend-without-rewriting-the-app*.

The harness is built for (1) first. (2) sharpens the safety and reliability story. (3) drives distribution. (4) is the eventual revenue.

## What Makes This Cool

The "whoa" moment is watching the router **hand off mid-task** between specialist models on a live arm:

> "Pick up the red block and stack it on the blue one."
> → Gemini Robotics plans the decomposition and identifies the grasp pose
> → Fine-tuned π0.5 executes the pick (fine motor control)
> → Classical motion planner executes the transit (deterministic, cheap)
> → π0.5 executes the place
> → Live panel shows each handoff, the router's reasoning, the trajectory each backend proposed, and the cost/latency profile of the run

Combined with trajectory ghosting — a translucent overlay rendered from the action chunk buffer, showing the next ~1s of planned motion ahead of where the arm actually is — this turns "the model is thinking" from a spinner into something you can watch happen in physical space.

The deeper insight: **completeness now comes from composition, not one giant model.** No single VLA is best at everything. Auto-mode makes that visible, and the harness makes it reproducible.

## Premises

These are assumed true. If any turn out false, the plan needs revision.

1. **Cloudflare Containers GPU access is available in the user's account** — to be confirmed in Week 1. Fallback: Modal or RunPod for inference, CF credits cover orchestration/storage.
2. **UFactory 850 works with LeRobot out of the box** — [LeRobot PR #493](https://github.com/huggingface/lerobot/pull/493) added first-party 850 support and was tested on two real units. This is the load-bearing assumption that unlocks the project.
3. **Fine-tuning π0.5 on teleop data produces a working policy on the 850 for 2–3 simple tasks** — Physical Intelligence's stated range is 1–20 hours. Realistic working assumption: 2–4 hours produces *some* working behavior on simple tasks, but reliable execution at demo quality likely needs 6–10 hours. The compression risk is real: data collection on a novel embodiment is slower than vendor marketing suggests, and a Week 2–3 overrun on data eats into the router build. Mitigation lives in the classical-planner fallback path (see Recommended Approach).
4. **The router's value can be demonstrated empirically** — there exists at least one task where auto-mode decomposition + dispatch beats the best single backend on success rate, latency, or cost. If this can't be shown, auto-mode is a UI flourish, not a contribution.
5. **The demo environment supports the cloud + Edge Agent split** — the demo machine has at least one USB3/GigE camera, a modern Python environment, and a stable ≥25Mbps connection with <100ms RTT to a Cloudflare PoP. Without this, the architecture degrades to a single-machine deployment (still viable, but the "hosted inference" story weakens).
6. **Chunked-action serving is acceptable for the target tasks** — pick-and-place, stacking, and pouring all tolerate open-loop chunks of 0.25–1s; we do not need true closed-loop control at >5Hz over the WAN. If a target task turns out to require tight closed-loop reaction (e.g., catching a falling object), it is out of scope.

## Approaches Considered

### Approach A: Sim-only, no real hardware

Skip the UFactory 850 entirely, build everything against LIBERO or ManiSkill simulation. Use openpi's existing LIBERO eval.

- **Pros**: Lowest risk, fastest path to a working demo, no fine-tuning needed, no teleop rig
- **Cons**: Demo video is far less compelling, you own the arm so not using it is wasted, no real-world generalization story, every other student project will be sim-only
- **Reuses**: openpi LIBERO eval scripts, existing π0 base checkpoints
- **Effort**: S
- **Risk**: Low

### Approach B: Real arm, classical control only

Use the UFactory 850 with classical motion planning (xArm SDK's built-in IK + MoveIt) and use VLMs (Gemini Robotics, Claude) only for high-level planning and grasp pose prediction. No VLA fine-tuning. The "models" you route between are different LLMs/VLMs doing planning, not different VLAs doing control.

- **Pros**: Reliable, fast to ship, no fine-tuning risk, real hardware demo, real Cloudflare hosting story for the planning layer
- **Cons**: Not actually a VLA harness — it's an LLM-planner harness. The core "agent harness for foundation models" framing weakens. Peer review will notice.
- **Reuses**: xArm Python SDK, Gemini Robotics API, classical planning libraries
- **Effort**: M
- **Risk**: Low

### Approach C: Real arm + fine-tuned VLA + router (RECOMMENDED)

Collect teleop data on the 850, fine-tune π0.5 via LoRA on that data, host the resulting checkpoint on Cloudflare Containers GPUs, proxy Gemini Robotics as a second backend, build the router/auto-mode + UI on top.

- **Pros**: Real CS153 contribution (you trained a model), genuine "VLA harness" story, real-arm demo, real hosted-inference business story, defensible in peer review
- **Cons**: Highest risk path — fine-tune could underperform, teleop data collection often overruns, more moving parts
- **Reuses**: openpi training stack, LeRobot UFactory 850 wrapper, Cloudflare Workers/Containers/R2/AI Gateway, xArm SDK
- **Effort**: L
- **Risk**: Medium-High

## Recommended Approach

**Approach C, deployed against the cloud + Edge Agent architecture (see Architecture section), with Approach B baked in as a per-skill fallback for the demo path.**

For every skill the fine-tuned π0.5 is supposed to execute, also implement a classical-planner version that runs locally in the Edge Agent. This serves three purposes:

1. **Demo insurance**: If the fine-tune underperforms on demo day, the classical path keeps the video shippable.
2. **Real research contribution**: Auto-mode now has a defensible decision to make per subtask — "learned policy vs classical planner, which has higher predicted success here?" This sidesteps the "did you really fine-tune a VLA in 10 weeks" peer-review question and turns it into a positive: you're studying when learned policies are worth the cost.
3. **Manufacturing credibility**: The classical path is *deterministic* and *auditable*, which is what manufacturing users care about. Auto-mode is then a way to opportunistically use VLAs where they're genuinely better, while leaving the safety floor at deterministic control.

## Architecture

The system splits cleanly along a **cloud / local seam**. Model inference, planning, telemetry, and session state live in the cloud. The real-time control loop and the arm itself live in a local **Edge Agent** process that runs on the user's own machine alongside their robot. This is the only architecture that meets a real-time control budget — pushing a 30Hz action loop over WAN to Cloudflare's edge and back would add 100–300ms RTT per step, which is fine for chunked actions but unworkable for closed-loop control.

The cloud serves the *model*. The Edge Agent runs the *loop*.

**Why three components in the cloud instead of one Worker:**

- **Planner Worker** is stateless and short-lived: take a task string + capability cards, call an LLM, return a plan DAG. Lives in a normal Worker (sub-30s CPU limit is plenty for a single LLM tool-call).
- **Dispatcher Durable Object** owns one run from start to finish: walks the DAG, multiplexes obs/action streams between the Edge Agent and the chosen backend per node, fans telemetry out to the UI, persists trajectory to R2. Durable Objects survive WebSocket-length sessions where a stateless Worker cannot.
- **Session DO + R2** stores history, structured run logs, model checkpoints, and run videos for replay and critic.

This split also means the Edge Agent only talks to the Dispatcher — one WebSocket, one auth boundary, one place to apply rate limiting, quotas, and billing.

**Router LLM choice:** **Claude Sonnet 4.6** via AI Gateway. Sonnet is the cost/quality sweet spot for tool-call routing: ~$3/M input tokens, sub-second first-token latency, strong tool-use evals, and prompt caching cuts the cost of re-running the same capability-card prompt to near-zero. Opus is overkill for this; Haiku starts to mis-route on adversarial plans. The router prompt fits comfortably under 4k tokens including capability cards, so cost-per-plan is well under $0.01. If Sonnet is unavailable (outage, quota), fall back to GPT-4.1 via the same AI Gateway.

**Trajectory ghosting** is rendered directly from the buffered action chunks: the Edge Agent forward-simulates the next N actions through the 850's kinematics and streams a sequence of TCP poses to the UI, which draws them as a fading polyline overlaid on the live camera feed. No additional model call required — it is a visualization of the chunk buffer the model already produced.

**Latency profile:**

- Plan generation (Planner Worker): 2–5s, once per task
- π0.5 chunk inference (CF Container): 80–200ms per chunk of 25–50 actions
- Network RTT, cloud ↔ Edge Agent: 30–100ms (Cloudflare PoP-dependent)
- Local control loop: 33ms cadence (30Hz), driven by Edge Agent from the chunk buffer
- End-to-end perceived latency, user prompt → arm starts moving (warm container): 3–6s
- End-to-end perceived latency on **cold start**: 8–25s. Mitigated by the pre-warm cron (Cost Model line item) for the demo workspace and the public sandbox; user-created workspaces accept cold-start as a first-run cost.

The chunked-action approach (π0-FAST and π0.5 emit multi-step action sequences per inference) is what makes cloud serving viable. Without it, this architecture would not be feasible.

**Phase-MVP vs Phase-Product overlay:** the diagram above describes the *Phase-Product* shape. In Phase-MVP, the Planner Worker, Dispatcher Durable Object, and Session DO are collapsed into a single Worker + one Durable Object holding run state. The Edge Agent's contract with the cloud is unchanged across phases — only the cloud-side internals consolidate. This is the "make the change easy, then make the easy change" sequence: design the seams now, refactor when there is a second concurrent user.

### Backend types and capability cards

Backends split into three roles. A single model can fill more than one role (e.g., Gemini Robotics can plan *and* act).

```python
class Planner(Protocol):
    """Produces a plan DAG from a task string + initial scene observation."""
    capability_card: CapabilityCard
    async def plan(self, task: str, scene: Observation) -> PlanDAG: ...

class Controller(Protocol):
    """Produces action chunks given streaming observations + per-node instruction."""
    capability_card: CapabilityCard
    async def act(self, obs_stream: AsyncIterator[Observation],
                  instruction: str) -> AsyncIterator[ActionChunk]: ...

class Critic(Protocol):
    """Reads a completed trajectory and emits an evaluation. Used post-hoc and as a
    passive monitor during execution."""
    capability_card: CapabilityCard
    async def critique(self, trajectory: Trajectory) -> CriticReport: ...
```

A **capability card** is what the router LLM reads to pick a backend. It is a structured tool description, not free-form text:

```yaml
id: pi05-ufactory-ft-v1
name: "π0.5 (fine-tuned on UFactory 850)"
roles: [controller]
embodiment:
  arm: ufactory-850
  dof: 6
  action_space: ee_pose_delta_base_frame    # see Open Questions #6
  control_rate_hz: 30                        # via chunked actions
input_modalities: [rgb_image, joint_state, language]
camera_views: [wrist, overhead_optional]
skills:
  - pick: {confidence: 0.8, learned_from: 240_demos}
  - place: {confidence: 0.85, learned_from: 240_demos}
  - stack: {confidence: 0.7, learned_from: 60_demos}
  - pour: {confidence: 0.4, learned_from: 0_demos}  # untrained → router should avoid
latency:
  p50_ms_per_chunk: 95
  p99_ms_per_chunk: 220
cost_per_chunk_usd: 0.0008
determinism: stochastic   # values: deterministic | stochastic | seeded
safety: {requires_envelope: true, supports_velocity_cap: true}
fallbacks: [classical-planner-pick, gemini-robotics-act]
```

The router LLM is shown the cards as tool descriptions. It produces a plan DAG annotating each node with `chosen_backend`, `reason`, and a `fallback_chain`.

### Fallback chain

Every node in the plan DAG has an explicit ordered fallback list. The Dispatcher walks the list on any of: HTTP error, timeout (>10s for an action chunk), safety stop, or critic-flagged trajectory deviation. Each fallback attempt is logged in the run record with the trigger reason.

Default fallback order for control nodes:
1. Primary chosen backend
2. The next-best controller with the relevant skill in its capability card (e.g., π0.5 base if fine-tuned fails)
3. Classical-planner backend in the Edge Agent (deterministic, lowest capability ceiling, highest reliability floor)
4. Hard stop with a critic-written report

The classical-planner step exists in the chain *even when the router didn't choose it*. This is the architectural commitment that makes the manufacturing-credibility story real.

**Bounded execution.** Two hard limits prevent infinite-loop pathologies:

- `max_attempts_per_node` (default 2) — each backend in a fallback list is tried at most this many times before falling through to the next entry. Without this bound, a flaky-but-not-down backend could be retried indefinitely.
- `max_wall_clock_per_run` (default 5 minutes, configurable per task) — once exhausted, the Dispatcher hard-stops the run regardless of remaining fallbacks. A run that runs out of time gets a `timeout` critic report, not a silent hang.

Both limits surface in the run record; both are workspace-configurable.

**State handoff at fallback boundaries.** When the Dispatcher walks from one backend to the next in the chain, the new backend cannot pick up "wherever the last one left off" — it needs an explicit handoff state. Mid-chunk failures are particularly bad because the arm may be in an unsafe or ambiguous pose. The handoff protocol:

1. **Edge Agent halts the arm at the next chunk boundary** (or immediately on safety event) and reports the current `RunState` to the Dispatcher.
2. `RunState` contains: current joint pose, current TCP pose, gripper state (open/closed/grasping), task-progress index in the plan DAG, last fully-completed chunk index, fresh observation (one frame from each camera), and a critic-written summary of what happened so far.
3. **Recovery primitives** (see Edge Agent description) may run before the new backend takes over: `home`, `open_gripper`, `relocalize`, or `abort_safely` — selected by the Dispatcher based on the failure type and the new backend's capability card.
4. The new backend receives the full `RunState` and the original task instruction, then plans its first chunk *as if starting from the recovered state*. Backends that cannot consume a partial-progress state (some VLAs require a "from scratch" prompt) signal this in their capability card and the Dispatcher invokes `home` + `relocalize` before handing off.

This is the architectural commitment that turns the fallback chain from "second-chance dispatch" into "graceful recovery." Without it, fallback degenerates to stop-and-restart, which Codex correctly flagged as the doc's most underspecified mechanism.

### Edge Agent ↔ Cloud version negotiation

The Edge Agent is pip-installable on the user's machine; Cloud-side components ship on whatever cadence FARM releases. Within weeks of release, in-the-wild Edge Agents will lag behind the Dispatcher. Without a protocol, this manifests as silent action-schema mismatches — the arm makes nonsense moves, the user has no idea why.

Every Edge Agent ↔ Dispatcher WebSocket handshake exchanges:

- `protocol_version` (semver): the wire protocol (obs/action chunk schemas, control messages). Bumped on breaking changes.
- `agent_version`: the Edge Agent package version (informational, surfaced in run records)
- `feature_flags`: optional capabilities the Edge Agent supports (e.g., trajectory ghosting render hooks, depth camera streaming)

Mismatched `protocol_version` causes the Dispatcher to reject the connection with a structured error message — "Edge Agent v1.0 detected, Dispatcher requires v1.2+; run `pip install -U farm-edge-agent`." The Edge Agent surfaces this in its CLI output verbatim.

Edge Agent supports an `--auto-update` flag (opt-in) that pip-upgrades itself on protocol mismatch and reconnects. Off by default for manufacturing users; on by default for the public sandbox.

### Network reality and offline mode

Premise 5 assumes ≥25Mbps and <100ms RTT. Reality for grad students on university WiFi behind captive portals and corporate proxies is worse. The DX has two affordances:

- **`farm doctor network`**: 30-second probe of DNS → WebSocket upgrade → RTT histogram → TLS chain → MTU → throughput. Emits a verdict (`OK / DEGRADED / FAILED`) with specific fix commands per failure mode.
- **`FARM_RELAY=on`**: tunnels WebSocket over HTTPS long-polling for environments that block WS upgrades. Higher latency, but it works behind aggressive proxies.
- **`farm run --offline`**: routes everything to the local classical-planner backend. No cloud calls. Lower capability ceiling, zero-network execution. First-class fallback, not an emergency hack.

### Reproducibility (`farm.lock`)

This is a **research-grade audit trail**, not a manufacturing-grade compliance audit. Be honest about what it is and isn't.

**It is**: a workspace can commit a `farm.lock` file:

```yaml
# farm.lock — committed alongside the experiment / pipeline definition
edge_agent: 1.2.3
protocol: 1.2
backends:
  - id: pi05-ufactory-ft-v1
    digest: sha256:9a3...
  - id: classical-planner
    version: 1.0
calibration_hash: sha256:a8b...
```

`farm verify <run-id>` checks that the run was executed against the locks in `farm.lock`. CI-friendly exit codes; failures point at the specific drift. Run records are HMAC-signed against the workspace's signing key so a holder of that key can verify the record was not tampered with after the fact.

**It is not**: real manufacturing audit. What FARM does *not* provide, and would need before a compliance-shaped pilot:

- Third-party-verifiable signing (HMAC requires the verifier to hold the key; this is an integrity check, not a public audit. Phase-Product upgrade: Cosign-style asymmetric signing — see TODOS.md).
- Proof that the arm actually executed the logged trajectory (would require independent hardware-side telemetry capture and cross-signing).
- Operator identity, approval trail, e-stop circuit state, environmental sensors, firmware hashes, safety PLC state.
- Calibration *correctness* attestation (the hash proves the file did not change; it does not prove the calibration is right).
- An immutable append-only log (R2 is workspace-mutable in Phase-MVP; immutable backing is Phase-Product).

Manufacturing users get research-grade reproducibility on Day 1 and a published gap list for what they need before a real production pilot. That is more honest than overpromising audit and getting caught at pilot time.

## Safety

This is a real industrial-class 7kg arm with non-trivial workspace and force. Safety is a hard requirement, not a polish item, and it is also a load-bearing piece of the manufacturing-user story. The Edge Agent is the safety boundary — no command reaches the arm without passing every check below.

- **Workspace envelope**: configurable per-task axis-aligned bounding box in Cartesian space. Any commanded pose outside the envelope is rejected and triggers a soft stop. Default envelope is conservative (~40cm cube centered in front of the arm).
- **Velocity and acceleration caps**: hard ceiling on joint velocity and TCP linear velocity. The Edge Agent rate-limits action chunks; any chunk requiring motion above the cap is clamped and logged.
- **Singularity and self-collision check**: every commanded waypoint is run through the xArm SDK's IK + collision check before execution. Failed checks trigger pause and a critic note.
- **Watchdog**: if the Dispatcher's WebSocket goes silent for >1s during execution, the Edge Agent halts the arm in place. The cloud cannot stall the arm in a dangerous configuration.
- **Physical e-stop**: the UFactory 850's hardware e-stop is wired through the Edge Agent's start sequence; the system refuses to begin a run if the e-stop circuit is not detected.
- **Human-in-the-loop run gate**: every autonomous run requires a UI confirmation before execution starts. Demo recordings happen with the operator in arm's reach.
- **Trailing critic**: a Critic LLM annotates the run record at each pause point (between chunks, at end of plan node, at run end). It is *not* a real-time safety boundary — LLM call latency (~500ms-2s) is too slow for that, and the next action chunk is already executing by the time a critic conclusion arrives. The critic catches failure *patterns* across a run for replay and learning; the Edge Agent's deterministic checks above catch failures *in the moment*. Don't conflate them.
- **Calibration drift detection**: the camera intrinsics + extrinsics file is hashed at calibration time, and that hash is included in every run record. The Edge Agent refuses to start a run if the calibration file's mtime is older than 24 hours and the user has not acknowledged it via a `--accept-calibration` CLI flag. Run records with mismatched calibration hashes are flagged in the UI so peer reviewers can see which runs share a calibration baseline.
- **Recovery primitives**: the Edge Agent exposes a small set of standard low-level recovery actions that the Dispatcher can invoke during fallback handoffs or after a critic-flagged anomaly. These are first-class capabilities, not opportunistic hacks:
  - `home`: move the arm to a known safe pose, gripper open. Deterministic via xArm SDK IK.
  - `open_gripper`: release any grasped object. Used before retrying a failed grasp.
  - `relocalize`: capture fresh frames from all cameras, run perception, return an updated `RunState`.
  - `retry_grasp`: re-attempt the last grasp from the current TCP pose. For slip recovery rather than planning errors.
  - `abort_safely`: descend to the nearest in-envelope waypoint, gripper open, watchdog disarmed. Terminal — the run ends.
  These primitives are what makes the fallback chain produce *graceful recovery* instead of *stop-and-restart*. They are exposed in the Python API for power users who want to compose them directly.

Safety failures are first-class events in the telemetry stream. They appear in the UI alongside model thinking and trajectory data, so a peer reviewer (or a manufacturing user) can audit *why* the arm stopped.

## Multi-Tenancy and Auth

This section describes the **Phase-Product** account model. **Phase-MVP** runs with a single API key in a config file and a single hardcoded workspace constant — no SSO, no quotas, no public sandbox. The interfaces below are designed so the Phase-Product implementation is incremental, not a rewrite.

Even in the CS153 build, the harness is account-shaped. Research labs and manufacturing users will not share a single endpoint.

- **Accounts**: Cloudflare Access (built in to the credit pool) handles SSO for the dashboard. A user belongs to one or more **workspaces**.
- **Workspace**: the unit of isolation. Owns a set of registered Edge Agents (by API key), a quota, a billing record, and a set of saved tasks. Runs are scoped to a workspace.
- **API keys**: each Edge Agent registers with a workspace-scoped key. Key rotation is one-click in the UI. Keys never leave the user's machine in plaintext after creation.
- **Public demo endpoint**: read-only sandbox workspace with rate limits (10 plans/hour, no real-arm dispatch — sim only). Lets peer reviewers click around without an account.
- **Quotas**: per-workspace daily compute budget. Default low; can be raised via a contact form in the demo build, automated billing later.

This is the minimum viable account model. It is not the full SaaS surface — but the architectural commitments (workspace-scoped tokens, per-workspace R2 prefixes, per-workspace AI Gateway tags for cost attribution) mean adding the rest later is incremental, not a rewrite.

## Observability

**Phase-MVP** writes run records as JSONL files locally (under `~/.farm/runs/<run-id>/`) and exports via `farm export` on demand. **Phase-Product** writes records to R2 alongside the run video and exposes them via the dashboard and Python API. The record schema is identical across phases; storage location is the only thing that changes.

Every run produces a **structured run record** stored in R2 alongside the run video (Phase-Product) or in `~/.farm/runs/` (Phase-MVP). The record is a JSON blob with:

- The original prompt and the plan DAG (including router reasoning per node)
- For each node: chosen backend, fallback events, every action chunk emitted (with timestamp), every observation seen (downsampled), every safety event, critic notes
- Wall-clock timing and cost breakdown
- A `replayable: true` flag and a content hash, so reruns of the same record are reproducible (when the chosen backend supports `determinism: seeded`)

Run records are downloadable as `.jsonl` and as LeRobot-format trajectory shards. **This is the killer feature for the research-lab user**: their experiment artifacts come pre-packaged for a paper appendix.

The dashboard surfaces:
- Live run telemetry (the trajectory ghost, model thinking, current backend, current safety state)
- Run history with diff between runs (what changed in the plan, what changed in outcomes)
- Aggregate metrics per backend (success rate by skill, latency distribution, cost-per-success)

## Evaluation

A single primary eval task anchors the project. Everything else is a variant.

**Primary task: colored-block pick and stack.** Two colored blocks (red and blue) placed in randomized positions within the workspace envelope. The arm must pick the red block and stack it on the blue block, regardless of starting positions.

Success criteria:
- ≥80% success rate over 20 trials per backend mode (manual π0.5, manual Gemini, manual classical, auto-mode)
- Failure is defined as: object knocked off the workspace, drop during transit, misstack (red not on blue), or safety stop
- Reported metrics: success rate, mean wall-clock time per attempt, $/successful-attempt (compute cost), and number of router handoffs in auto-mode

**Secondary tasks** (one or two if time allows, in priority order):

1. **Tool-tray kitting** (manufacturing-flavored) — given a tray of mixed parts (3 distinct SKUs, e.g., bolt, washer, nut), sort each into its labeled bin. Tests sustained execution over many sub-pick-and-places, classification under varying lighting, and recovery from a missed grasp. This is the task that earns the manufacturing-persona credibility — kitting is a real bench task in low-volume assembly. Reports per-SKU success rate, total cycle time, and rate of safety stops.
2. **Pour-from-cup-to-cup** — tests trajectory smoothness, grasp orientation under wrist constraints, and tolerance for spilled liquid as a failure mode. Visually compelling for the demo video.
3. **Towel fold** — deformable handling; genuinely hard for VLAs in 2026. Almost guaranteed to fail. Included because reporting an honest failure on a hard benchmark is more credible than only reporting easy wins.

The success criteria for secondary tasks are honest reporting of pass/fail per trial, not a threshold to defend. The kitting task is the highest-priority secondary because it is the one a manufacturing peer reviewer will look for.

**Why colored-block-stacking as primary**: it is well-trodden in the VLA literature (so baselines exist for comparison), it forces real perception (color discrimination + position generalization), it benefits from VLA generalization (the classical planner cannot easily handle "red" without color detection bolted on), and it is short enough that 20 trials fit in a single afternoon.

**Why kitting as the highest-priority secondary**: it is the simplest task that makes a manufacturing user nod. It is also a stress test for the fallback chain: many sequential picks means many opportunities for a single bad chunk to require the classical-planner fallback to recover gracefully without dropping the part.

### Supported task class

FARM's cloud + Edge Agent architecture, with chunked-action serving at 1–4 Hz, is sound for **quasi-static tabletop manipulation**: tasks where 0.25–1s of stale observations between chunks is tolerable. The eval tasks above all fit this class. Explicitly **out of scope** for Phase-MVP and Phase-Product:

- **Fast-reactive tasks** that need closed-loop reaction at >5 Hz (catching a falling object, reactive collision avoidance against a moving obstacle). The WAN round-trip floor makes this physically impossible regardless of model quality.
- **Contact-rich insertion** requiring force feedback (peg-in-hole at sub-mm tolerance, bolt threading, snap-fit assembly). No force/torque sensing in the perception stack.
- **Slipping-grasp recovery within a single chunk**. If a grasp slips mid-chunk, recovery happens at the next chunk boundary (≥250ms later), which is too slow for some objects.
- **Bimanual coordination**. Single-arm only in Phase-MVP and Phase-Product.

Saying this explicitly is part of the manufacturing-credibility story: a manufacturing engineer evaluating FARM needs to know which of their tasks are candidates and which are not, before the pilot conversation gets to the "why does it not work for our assembly line" stage.

### Trial scoring (ground-truth method)

The 80%-success-rate target needs an unambiguous judging method. Phase-MVP uses **operator-labeled trials with video evidence**: each trial is recorded; an operator marks pass/fail at run end against a written success rubric (e.g., "red block fully resting on blue block, not knocked off, no parts dropped"); the label is committed to the run record alongside the video shard so a peer reviewer can audit any labeled trial.

A simple automated heuristic runs in parallel for sanity-check: post-run, the wrist camera frame is checked for the target object configuration via color thresholding + rough geometry. Disagreements between operator and heuristic are flagged in the run record but do not change the labeled outcome. Camera-based auto-judging without operator review is a Phase-Product goal.

## Fine-Tuning Plan

The contribution depends on the π0.5 fine-tune actually working on the UFactory 850. This section commits to the training-loop specifics that make the result reproducible, not vibes.

**Dataset target.** Per task: **200–400 demonstration episodes**, each 5–15 seconds long, collected via teleoperation. Total wall-clock for the primary task (colored-block stack): 4–8 hours of clean teleop, plus 2–3 hours of fixturing, lighting setup, and bad-take re-recording. Secondary tasks: 100–200 episodes each if time allows.

**Episode schema** (LeRobot-format):
- Wrist camera frame (RGB, 224×224, JPEG-encoded, 10 Hz)
- Overhead camera frame (optional, same format)
- Joint state (6-DOF position + velocity, gripper open/close)
- End-effector pose in base frame (TCP position + quaternion)
- Action: end-effector pose delta in base frame at the next timestep (matches `action_space: ee_pose_delta_base_frame` in the capability card)
- Language instruction (canonical per task, e.g., "pick the red block and stack it on the blue block")
- Episode metadata: calibration hash, operator ID, success flag (labeled at collection time)

**Train / validation / test split.** 80 / 10 / 10, **held out by collection session** (not by episode). Holding out by session prevents leakage from lighting and fixturing correlations that would otherwise inflate validation accuracy.

**Hyperparameters** (LoRA fine-tune on top of frozen π0.5 base):
- LoRA rank: 16 (start), 32 if undertrained at rank 16
- Learning rate: 1e-4, cosine decay over training
- Batch size: 8 episodes
- Epochs: 5–10, with eval-based early stopping
- Optimizer: AdamW with weight decay 0.01
- Action loss: flow-matching MSE on the action tokens (π0.5's native loss)

**Eval cadence.** Every 500 training steps:
- 20 rollouts in LIBERO-style sim against held-out scenes (cheap, fast, statistically meaningful)
- Sim success rate is the checkpoint-selection metric, NOT lowest training loss

**Checkpoint selection.** The checkpoint with the highest sim-eval success rate, not the last-epoch checkpoint. Saved alongside the eval curve so the choice is auditable.

**Rollback criteria.** If no checkpoint clears **50% sim success rate** on the held-out tasks by end of training, the fine-tune is rejected. Causes to investigate, in priority order: dataset too small, dataset too noisy (bad demos), action-space mismatch, calibration drift between collection sessions, model architecture mismatch with the 850's joint count. Decision tree for each cause is shipped as `docs/training-runbook.md`.

**Training compute.** Modal or Lambda Labs (NOT Cloudflare — fine-tuning is not credit-eligible). One-shot Week 3; budget $300–1000 total. Repeated runs on hyperparameter sweeps add to this.

**Reproducibility.** The full training command, dataset version, base-model digest, and hyperparameter file are pinned in the fine-tuned checkpoint's model card on Hugging Face. Re-running the same command on the same dataset must produce a checkpoint within ε of the reported eval numbers.

## Success Criteria

In priority order. Each is binary (yes/no) at submission time.

1. **Demo video shows a real UFactory 850 executing a multi-step task via auto-mode**, with at least one visible handoff between two different backends (e.g., π0.5 → classical planner, or Gemini → π0.5). At least one safety event is visible in the run, with the system handling it gracefully. **Demo design must guarantee a safety event fires** — easiest path: place a target object near the edge of the workspace envelope so a planned overshoot trips the soft-bound check, recovers, and re-plans. A demo with no safety event makes the safety system invisible to the viewer.
2. **The hosted inference path actually works end-to-end**: a peer reviewer can install the Edge Agent on their own machine, point it at the public demo endpoint, and execute a recorded plan against a simulated arm in <15 minutes. 3. **At least one quantitative result on the primary eval task** showing auto-mode performance against each single-backend baseline across ≥20 trials per mode. Report success rate, mean time, and $/successful-attempt. Auto-mode does not have to win every metric — but the result must be reported honestly with a CI.
4. **Code is reproducible**: README walks a peer reviewer from `git clone` to running the harness in sim mode (no arm) in <30 minutes. Real-arm reproduction is a stretch goal — too much hardware setup to expect from reviewers.
5. **At least the primary eval task plus the kitting secondary** demonstrated on the real arm with reported metrics. Additional secondaries (pouring, folding) are nice-to-have.
6. **Run records are downloadable** as both `.jsonl` and LeRobot-format shards from the dashboard. This is what the research-lab persona needs.

## Not in Scope

Explicit deferrals. Every item here was considered and deliberately excluded from the Phase-MVP build that ships for CS153. Each has a one-line rationale.

- **Real-time closed-loop control at >5Hz over WAN** — Premise 6. Chunked-action serving is what makes cloud-hosted VLAs feasible at all; tighter loops require local model serving, which is a different architecture.
- **Multi-tenant concurrent execution** — workspace data model exists in the design, but Phase-MVP runs one user (you). Concurrent dispatches, GPU multiplexing, and quota enforcement are Phase-Product.
- **Bring-your-own-model registration API** — Phase-Product. The capability-card *format* is fixed in Phase-MVP, but only FARM-hosted backends register against it.
- **Multi-embodiment beyond UFactory 850** — drivers for Franka, UR5, SO-ARM live in the Edge Agent design (the `lerobot-mock` driver is a stub showing the contract), but only the 850 driver is shipped and tested.
- **Bit-exact reproducibility across hardware** — see Reproducible Mode stretch. Honest claim is "same approximate trajectory, same plan DAG." Bit-exactness is not achievable with flow-based VLAs.
- **Real-time critic anomaly detection** — see Safety. The critic produces trailing annotations; real-time safety is the Edge Agent's deterministic checks, not the critic.
- **Voice input** — dropped from earlier stretch goal list. Visual demo is sufficient.
- **Billing / metering enforcement** — quota *caps* exist (hard daily limits); usage *metering* and any kind of charging is Phase-Product.
- **GPU container auto-scaling under load** — single warm container in Phase-MVP. Scaling policy is Phase-Product.
- **The full Planner / Dispatcher / Session split** — the *design* names the seams; the Phase-MVP *implementation* collapses them into a single Worker + one Durable Object for run state. The split is incremental refactoring, not a rewrite.
- **Cross-embodiment auto-routing** — selecting which *arm* to dispatch to, not just which model. Phase-Product, listed as a stretch.
- **A polished documentation site** — `docs/` directory in the repo gets a getting-started guide; a full docs site is Phase-Product.

## Stretch Goals (only if ahead of schedule)

- **Bring-your-own-model** — external developers register an HTTP endpoint with a capability card; the router considers it alongside hosted backends. Trivial to implement, huge ecosystem story.
- **Reproducible mode** — given the same prompt, seed, and scene observation, the *plan DAG* and *backend assignments* are identical, the *classical* legs of execution are bit-exact, and the *VLA* legs are seeded-stochastic (same approximate trajectory, small numerical drift from GPU non-determinism). Bit-exact end-to-end reproduction across hardware is explicitly out of scope — it is not achievable with flow-based VLAs on heterogeneous GPUs. This is enough for replayable research artifacts in practice; "same outcome distribution" is the honest claim.
- **GR00T N1 as a third controller** — adds variety to the auto-mode story and gives the router a third option to choose between.
- **Sim-twin per workspace** — every real-arm run also executes in a Mujoco / Isaac Sim mirror. The UI shows them side by side. Researchers will pay for this alone.
- **Cross-embodiment auto-route** — same task, two different arms registered to the same workspace; auto-mode picks not just the model but the arm. Demonstrates the harness is actually embodiment-agnostic.
- **A second user's robot pointed at the public endpoint** — even a friend's arm running against your hosted π0.5 is a huge proof point for the hosting story. The cheapest path to a "users beyond me" claim.
- **CSV/Notebook export** — one-click export of a run record set into a Jupyter notebook with all the metrics pre-loaded. Research-lab killer feature.

## Related Work

The harness layer sits between several active efforts. The differentiation is sharper when stated against them by name:

- **LeRobot (Hugging Face)** — the most overlapping project. LeRobot is a *library* for training and running policies on real arms, including UFactory 850 (PR #493). It is not a hosted runtime, has no router, no per-action audit trail, no fallback chain. This project depends on LeRobot for data collection and for the 850 wrapper. It does not compete with it.
- **openpi (Physical Intelligence)** — the reference implementation for π0/π0.5 training and inference. Provides the model and training stack; provides no harness, UI, multi-tenancy, or routing. This project depends on openpi for fine-tuning and for the inference server image.
- **NVIDIA Isaac / GR00T platform** — vertically integrated stack from training to deployment, NVIDIA-hardware-centric. Different audience (large industrial customers); not a Cloudflare-credit-shaped offering.
- **Robotic operating systems (ROS 2, MoveIt)** — orthogonal. Classical planning and middleware. The classical-planner backend uses these. The harness sits *above* ROS, not against it.
- **Foundation model APIs (Gemini Robotics, etc.)** — point solutions. Each ships its own SDK and assumes its own model. This project's wedge is exactly the harness that lets you switch between them — and switch *between* them and classical control — without rewriting the application.

The differentiating wedge: **the harness is the only layer where auto-routing between learned and classical control is first-class, with a structured audit trail per action.** That phrasing is defensible against every project listed above.
